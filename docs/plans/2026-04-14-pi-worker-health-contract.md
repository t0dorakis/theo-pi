# Pi Worker Health Contract

## Goal

Define the machine-readable and operator-readable health contract for Theo’s supervised Pi worker.

The health contract should be simple enough for local VM operations, useful enough for future automation, and stable enough to back future gateways or wake-hook integrations without redesigning the worker.

## Primary Health Fields

Suggested health payload:

```json
{
  "ok": true,
  "daemonStatus": "running",
  "sessionName": "theo-pi",
  "workspacePath": "/home/piagent/workspaces/theo-pi",
  "pid": 12345,
  "restartCount": 1,
  "lastHeartbeatAt": "2026-04-14T12:11:35Z",
  "lastSuccessAt": "2026-04-14T12:10:54Z",
  "bootstrapVersion": "2026-04-14.1",
  "notes": []
}
```

## Field Definitions

### `ok`
Boolean summary for whether the worker is healthy enough for normal operation.

### `daemonStatus`
String status of the supervised Pi runtime.

Allowed initial values:
- `starting`
- `running`
- `stopped`
- `failed`
- `stale`

### `sessionName`
Named `tmux` or runtime session currently associated with this worker context.

### `workspacePath`
Absolute path to the active workspace under bounded worker directories.

### `pid`
Process ID of Pi runtime if known. May be `null` when stopped or unknown.

### `restartCount`
Count of restart attempts since the last stable reset or explicit operator reset.

### `lastHeartbeatAt`
Timestamp of most recent successful heartbeat update.

### `lastSuccessAt`
Timestamp of most recent known good worker action, such as successful startup or successful handled prompt.

### `bootstrapVersion`
Version or generation marker for the worker bootstrap/runtime spec.

### `notes`
Array of human-readable warnings or observations. Examples:
- `"heartbeat stale"`
- `"restart cap nearly reached"`
- `"workspace path missing"`

## Output Forms

## 1. Machine-readable JSON

Primary canonical form. Stored in:

```text
~/.pi-worker/health.json
```

This is the source for scripts, future gateways, and status tooling.

## 2. Human-readable CLI output

Example:

```text
Worker: healthy
Session: theo-pi
Workspace: /home/piagent/workspaces/theo-pi
Status: running
PID: 12345
Restarts: 1
Last heartbeat: 2026-04-14T12:11:35Z
Last success: 2026-04-14T12:10:54Z
```

This can be generated by `pi-worker-status` or equivalent.

## Health Evaluation Rules

## Healthy (`ok: true`)

Should require at least:
- `daemonStatus === "running"`
- heartbeat is fresh enough
- workspace path exists
- required state files are writable/readable

## Unhealthy (`ok: false`)

Should result from any of:
- `daemonStatus === "failed"`
- heartbeat stale beyond threshold
- workspace missing
- runtime PID missing unexpectedly while status claims running
- repeated restart failures beyond cap

## Stale rules

Suggested initial stale threshold:
- heartbeat older than 2 expected intervals, or
- heartbeat older than a fixed operator-defined threshold such as 2–5 minutes

This threshold should be configurable later, but fixed is acceptable in v1.

## Success semantics

`lastSuccessAt` should be updated on meaningful success events, such as:
- Pi started successfully
- restart completed successfully
- health checks completed without error
- future: a prompt/run completed successfully

## Notes and Warnings

`notes` should be used for degraded-but-not-failed states, such as:
- stale heartbeat but recent logs still present
- restart count elevated
- bootstrap version mismatch
- session exists but Pi PID missing

This allows a warning path between “healthy” and “failed.”

## Future Extension Fields

Reserve room for later additions such as:
- `channelStatus`
- `gatewayStatus`
- `wakeQueueDepth`
- `checkpointLabel`
- `provider`
- `workspaceBackend`
- `lastWakeSignalAt`

These should be optional and must not break the base contract.

## Operator Usage

Theo should be able to answer these questions from health output alone:

1. Is the worker healthy enough to trust?
2. Which workspace/session is active?
3. Has the worker restarted recently?
4. Is the worker stale or actually failed?
5. Do I need to inspect logs or restart now?

## Non-Goals

The initial health contract does **not** need to:
- report deep internal Pi model/tool state
- expose browser/web UI compatibility
- carry full historical metrics
- replace logs as the source of detailed failure reasoning

## Acceptance Criteria

The health contract is successful if:
- it is easy to read over SSH
- it is easy to consume from scripts
- it distinguishes healthy, stale, stopped, and failed states
- it includes enough metadata to guide restart and recovery decisions
